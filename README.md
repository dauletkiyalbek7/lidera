# Lidera

Мульти-проектная платформа управления рекламными проектами.
Полное ТЗ — в [CLAUDE.md](CLAUDE.md); этот файл только про запуск, структуру кода и деплой.

## Стек

Next.js 16 (App Router) · TypeScript · Tailwind CSS 4 · Supabase (Postgres + Auth + RLS) · Vercel.

## Запуск

```bash
npm install
cp .env.example .env.local   # заполнить значениями своего проекта Supabase
npm run dev
```

Переменные окружения:

| Переменная | Где используется |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | сервер; уходит в браузер |
| `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` | сервер; уходит в браузер, доступ ограничен RLS |
| `SUPABASE_SERVICE_ROLE_KEY` | **только сервер**, обходит RLS: аккаунты сотрудников, вебхуки, токены |
| `LIDERA_SECRETS_KEY` | **только сервер**, 32 байта в base64: шифрование токенов интеграций |

Ключ шифрования генерируется так:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
```

Сменить `LIDERA_SECRETS_KEY` после запуска нельзя без потери сохранённых токенов:
старые расшифровать станет нечем, интеграции придётся подключать заново.

## База данных

Миграции в `supabase/migrations/`, применяются строго по порядку:

| Файл | Что делает |
|---|---|
| `0001_schema.sql` | таблицы по разделу 8 ТЗ |
| `0002_rls.sql` | функции доступа и RLS-политики на всех таблицах |
| `0003_profile_trigger.sql` | автосоздание `profiles` при создании пользователя |
| `0004_function_grants.sql` | хелперы RLS недоступны анонимному пользователю |
| `0005_write_policies.sql` | точечные права на запись: РОП, директор, участник |
| `0006_integration_secrets.sql` | зашифрованные токены интеграций (RLS без политик) |
| `0007_integrations_unique.sql` | одно подключение на провайдера в проекте |
| `0008_hr.sql` | посещаемость, график, правила зарплат, договоры |
| `0009_ads.sql` | кампании и дневные метрики рекламы, курс валюты кабинета |
| `0010_creatives.sql` | креативы, привязка лидов и продаж, метрики по объявлениям |
| `0011_lead_intake.sql` | приём заявок с сайта (RLS без политик) |
| `0012_saved_funnels.sql` | сохранённые рабочие воронки Marketing Dashboard |
| `0013_chat.sql` | чат-бот: переписки, сообщения, секрет вебхука |
| `0014_telegram.sql` | Telegram-бот: привязки сотрудников, секрет вебхука |

`supabase/seed/owner.sql` — создание первого владельца платформы (публичной регистрации нет).

### Изоляция проектов

RLS включён на всех таблицах с `project_id`. Владелец платформы (`profiles.global_role = 'owner'`)
и владелец проекта видят и меняют всё; активный участник проекта — только читает,
плюс точечные права на запись из `0005`. Чужой проект не возвращается из базы вообще,
а не прячется в интерфейсе.

Четыре таблицы — `integration_secrets`, `lead_intake`, `chat_webhooks`, `telegram_webhooks` —
идут с включённым RLS и **без единой политики**. Это не забывчивость: доступа к ним нет
ни у кого, кроме сервера с сервисным ключом. Токены и секреты вебхуков не должен видеть
даже владелец из браузера.

## Структура

```
src/
  app/
    login/                     вход по email и паролю
    projects/                  портал «Мои проекты» + создание проекта
    api/
      intake/                  приём заявок с сайта, лендинга, Tilda
      webhooks/chatplace/      входящие сообщения чат-бота
      webhooks/telegram/       привязка сотрудников к боту
    p/[projectId]/             кабинет проекта
      page.tsx                 Главная (дашборд)
      [...segments]/           заглушки разделов, у которых логика впереди
  components/
    layout/                    каркас кабинета, боковое меню, шапка
    metrics/                   карточка метрики, график, воронка, топы
    ui/                        кнопки, бейджи, иконки, таблица
  lib/
    supabase/                  клиенты: серверный и административный
    actions/                   серверные действия (формы)
    queries/                   чтение данных для страниц
    navigation.ts              карта блоков и разделов (ТЗ, раздел 7)
    metrics.ts                 производные метрики — только здесь
    format.ts                  деньги, проценты и даты — только здесь
    date-range.ts              общий выбор периода (ТЗ, раздел 5)
    crypto.ts                  шифрование секретов интеграций
    domain.ts                  роли, ниши, статусы, воронки
```

Доступ к Supabase — только на сервере (ТЗ, раздел 12). Браузерного клиента в проекте нет
намеренно: данные читаются в серверных компонентах, пишутся серверными действиями.

Новый раздел добавляется в `src/lib/navigation.ts`: он сразу появляется в меню
и получает заглушку. Реальная страница кладётся в `src/app/p/[projectId]/<путь>/page.tsx`
и перекрывает заглушку.

## Деплой на Vercel

### 1. Проект

Подключить репозиторий, framework определится сам (Next.js). Ветка `main` → production.
Node 20.9+ (зафиксирован в `engines`).

### 2. Переменные окружения

Задать все четыре из таблицы выше в **Production** (и в Preview, если превью нужны).
`SUPABASE_SERVICE_ROLE_KEY` и `LIDERA_SECRETS_KEY` — Sensitive, без префикса `NEXT_PUBLIC_`.

Проверка после сборки: ни один из двух серверных ключей не должен встречаться
в `.next/static`:

```bash
grep -rl "$(grep '^SUPABASE_SERVICE_ROLE_KEY=' .env.local | cut -d= -f2-)" .next/static | wc -l
```

Ноль — как и должно быть.

### 3. Deployment Protection — главная ловушка

Vercel по умолчанию закрывает превью-деплои авторизацией. Если включить защиту
на production, входящие запросы **не дойдут** до маршрутов:

- `/api/intake` — заявки с сайта и Tilda,
- `/api/webhooks/chatplace` — сообщения чат-бота,
- `/api/webhooks/telegram` — привязка сотрудников.

Meta, ChatPlace и Telegram получат в ответ HTML страницы входа вместо JSON и будут
считать интеграцию сломанной. Либо не включать защиту на production, либо добавить
эти три пути в Protection Bypass.

### 4. Supabase

- **Auth → URL Configuration → Site URL** заменить на боевой адрес.
- Миграции применить в том же порядке; на боевой базе — до первого захода пользователей.
- Превью-деплои по умолчанию смотрят в ту же базу. Если превью будут показывать
  посторонним, заводить отдельный проект Supabase, а не делить боевой.

### 5. После первого деплоя

Адреса вебхуков платформа показывает сама — они берутся из адреса запроса и станут
боевыми автоматически. А вот секреты выпускаются заново:

- «Ресурсы и воронки» → выпустить токен приёма заявок, вставить в сайт и Tilda;
- «Чат-бот» → выпустить токен, вставить в ChatPlace;
- «Настройки → Telegram-бот» → выпустить секрет и вызвать `setWebhook` (команда на экране);
- «Интеграции» → заново вставить токены Meta, TikTok, ChatPlace, Telegram, AI.

Токены из локальной базы на боевой не подойдут, если у боевого сервера другой
`LIDERA_SECRETS_KEY`.
